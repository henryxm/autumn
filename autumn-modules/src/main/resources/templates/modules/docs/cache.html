缓存体系（Cache）

EhCacheRedis 二级缓存失效广播运维接口

功能入口

  • 缓存核心服务：cn.org.autumn.service.CacheService
  • 共享缓存抽象：cn.org.autumn.service.ShareCacheService（BaseService 继承）
  • 运维控制器：/sys/cache/*（CacheController）
  • 失效同步通道：cache:invalidation（Redis Pub/Sub）

核心能力说明

  • 两级缓存：先查本地 EhCache，未命中再查 Redis。
  • 跨实例一致性：写入/删除后发布失效消息，其他实例删除本地缓存键。
  • 空值缓存：通过 NULL_PLACEHOLDER 支持缓存 null，防止缓存穿透。
  • 类型安全：根据 CacheConfig 对 key 类型进行转换，避免 Long/String 类型不一致。

关键配置项

配置来源 字段 默认值/策略 说明
CacheConfig expire / unit 1440 + MINUTES 本地缓存过期时间（当未注册配置且动态创建时）
CacheConfig redis / unit 1440 + MINUTES Redis 二级缓存过期时间
CacheConfig max 10000 最大条目数（动态创建配置时）
CacheConfig Null false 是否缓存 null 值
ShareCacheService getShareCacheExpire() 10（分钟） 共享缓存默认过期策略（可被子类覆盖）

API 示例（真实路径）

GET /sys/cache/status GET /sys/cache/list GET /sys/cache/keys/{name}?page=1&size=20 GET /sys/cache/search/{name}?pattern=*&page=1&size=20 GET /sys/cache/value/{name}?key=someKey DELETE /sys/cache/key/{name}?key=someKey DELETE /sys/cache/keys/{name} Body: ["k1","k2"] DELETE /sys/cache/clear/{name} DELETE /sys/cache/clear/all

返回结构示例

{"code":0,"msg":"success","data":{"key":"user:1","value":{"id":1},"valueType":"cn.org.autumn.modules.sys.entity.SysUserEntity","valueJson":"{\"id\":1}"}}

服务端调用示例（ShareCacheService / BaseService）

// 读取：未命中时自动回源并写入缓存 UserDto dto = getShareCache("user_profile", UserDto.class, userUuid, () -> loadFromDb(userUuid)); // 写入 putShareCache("user_profile", UserDto.class, userUuid, dto); // 删除（同时删单值+列表缓存） removeShareCacheAll("user_profile", UserDto.class, userUuid);

BaseCacheService 重点：getCache / getEntity 调用链

// BaseCacheService#getCache(Object key) 核心逻辑 // 1) 校验 @Cache(unique=true) // 2) 构建缓存 key // 3) compute(...) 缓存未命中时调用 getEntity(key) public T getCache(Object key) { checkFieldCacheUnique(true); Object cacheKey = buildCacheKey(key); return compute(cacheKey, () -> getEntity(key), getConfig()); } // BaseCacheService#getEntity(Object key) 默认实现 public T getEntity(Object key) { return getNameEntity("", key); }
  • getCache* 系列是统一入口：优先读缓存，未命中才回源加载。
  • getEntity* 系列是“回源策略”入口：默认通过注解+反射组装查询条件并查库。
  • 因此：不重写也可用（依赖默认实现）；有复杂业务时可重写 getEntity / getNameEntity 精准控制回源。

默认实现方式（注解 + 反射）

  • 单字段模式：在字段上标注 @Cache，默认实现会定位该字段并按字段值查询。
  • 复合 key 模式：在类上使用 @Cache(value={"f1","f2"}) 或 @Caches，按字段顺序/名称构造条件查询。
  • 命名缓存模式：通过 name 区分多个缓存定义，使用 getNameCache(name,...) / getNameEntity(name,...)。
  • 唯一性约束：getCache* 会校验 unique=true；非唯一字段应使用 getListCache*。
  • 自动创建：默认 getEntity* 仅在注解 create=true 时未命中可自动建实体；getCached/getEntitied 系列是“强制可创建”语义。

BaseCacheService 配置接口（可在实现类覆写）

BaseCacheService 提供了一组可覆写配置方法，支持按“默认 / 命名缓存 / 类型 / 类型+命名”四个维度定制缓存策略。

配置函数 默认值 作用
getCacheExpire* 10 本地缓存过期时间
getRedisExpire* 3 * getCacheExpire* Redis 二级缓存过期时间（默认是本地 3 倍）
getCacheMax* 1000 本地缓存最大条目数
getCacheUnit* TimeUnit.MINUTES 过期时间单位
isCacheNull* false 是否缓存 null 值
isCachePersistent* false 是否启用持久化（磁盘）
getCachePath* <#noparse>${user.home}/ehcache 持久化目录

方法使用决策图（getCache / getEntity / getCached / getListCache / getNameCache）

开始 ├─ 你是“读缓存入口”吗？ │ ├─ 是 │ │ ├─ 是否需要按 name 区分多套缓存？ │ │ │ ├─ 是 -> 用 getNameCache(name, ...) │ │ │ └─ 否 │ │ │ ├─ 字段是否 unique=false（一对多）？ │ │ │ │ ├─ 是 -> 用 getListCache(...) │ │ │ │ └─ 否 │ │ │ │ ├─ 未命中时是否要“强制创建”实体？ │ │ │ │ │ ├─ 是 -> 用 getCached(...) │ │ │ │ │ └─ 否 -> 用 getCache(...) │ └─ 否（你在实现回源逻辑） │ ├─ 标准回源（注解+反射）-> 用默认 getEntity(...) │ └─ 复杂回源（联表/状态过滤/租户隔离）-> 在实现类覆盖 getEntity(...) / getNameEntity(...)
方法 典型场景 未命中行为
getCache(key) 唯一键读缓存（默认入口） 自动调用 getEntity(key) 回源并写缓存
getCached(key) 唯一键读缓存且必须返回实体 未命中时尝试创建实体（强制创建语义）
getListCache(key) 非唯一字段（一对多列表） 自动调用 getListEntity(key) 回源列表
getNameCache(name, ...) 同一 Service 多套命名缓存 按 name 选择对应注解定义并回源
getEntity(...) 回源实现点（非缓存入口） 默认注解+反射查询；可由实现类覆盖
  • 一句话记忆：先选“缓存入口方法”（getCache/getListCache/getNameCache/getCached），再决定是否覆盖 getEntity 回源策略。

覆写示例（按业务场景调优）

public class UserCacheService extends BaseCacheService<UserDao, UserEntity> { // 默认缓存：15分钟 @Override public long getCacheExpire() { return 15; } // 命名缓存：黑名单缓存更短，避免脏数据滞留 @Override public long getCacheExpire(String naming) { if ("blacklist".equals(naming)) { return 2; } return 15; } // Redis 设置更长，降低跨实例回源 @Override public long getRedisExpire() { return 60; } // 热点不存在用户缓存空值，防止穿透 @Override public boolean isCacheNull(String naming) { return "user_profile".equals(naming); } // 开启持久化并指定目录 @Override public boolean isCachePersistent() { return true; } @Override public String getCachePath() { return "/data/cache/ehcache"; } }
  • 优先覆写“最小范围”函数：如只对某个命名缓存生效，优先覆写 getCacheExpire(String naming)。
  • 建议保持 Redis 过期时间大于本地过期时间，减少多实例重复回源。
  • 仅对高价值“空查询热点”开启 isCacheNull*，避免无意义占用。
  • 开启持久化时，请确保目录可写且有容量监控。

配置接口选择矩阵（覆写哪一个）

需求场景 推荐覆写接口 说明
整个 Service 的缓存策略统一调整 getCacheExpire() / getRedisExpire() / isCacheNull() 最简单，影响该 Service 下所有缓存
仅某个命名缓存策略不同 getCacheExpire(String naming) / isCacheNull(String naming) 按缓存名精细化控制，推荐优先使用
按返回类型/模型类型区分缓存策略 getCacheExpire(Class<X> clazz) / getCacheMax(Class<X> clazz) 适合同一业务中 DTO/VO/实体多类型缓存并存
同一类型下还要按 naming 再细分 getCacheExpire(Class<X> clazz, String naming) 粒度最细，适合复杂场景，注意可维护性
仅磁盘持久化策略需要差异 isCachePersistent* + getCachePath* 建议与容量监控、磁盘清理策略配套

选型顺序建议

  1. 先用默认版（无参数）跑通，保证策略简单统一。
  2. 出现单个缓存特殊需求时，再引入 String naming 版本。
  3. 只有在“同缓存名多类型策略冲突”时再引入 Class<X> 版本。
  4. 避免一开始就使用 Class<X> + naming，否则配置复杂度会快速升高。

最小覆写模板（推荐）

public class DemoService extends BaseCacheService<DemoDao, DemoEntity> { // 1) 全局默认 @Override public long getCacheExpire() { return 10; } // 2) 仅命名缓存差异 @Override public long getCacheExpire(String naming) { if ("hot_user".equals(naming)) return 2; return getCacheExpire(); } // 3) 仅在必要时按类型细分 @Override public <X> long getCacheExpire(Class<X> clazz) { if (clazz != null && "UserDTO".equals(clazz.getSimpleName())) return 5; return getCacheExpire(); } }

实现类覆盖建议

public class UserService extends BaseCacheService<UserDao, UserEntity> { // 覆盖回源逻辑：复杂查询、联表、过滤逻辑可放这里 @Override public UserEntity getEntity(Object key) { String userUuid = String.valueOf(key); return baseMapper.selectByUuidAndStatus(userUuid, 1); } }
  • 推荐只覆盖回源方法，不改 getCache* 主流程，保持缓存策略统一。
  • 若覆盖了 getEntity(Object... keys)，请保持参数顺序与缓存 key 构建规则一致。
  • 对于多套 key 方案，优先使用 getNameEntity(name,...) 明确语义，避免混用。

缓存接入模板（建议直接复用）

// 示例：在业务 Service 中统一定义缓存名和失效策略 public class UserProfileService extends BaseService<UserMapper, UserEntity> { private static final String CACHE_USER_PROFILE = "user_profile"; // 统一读取入口（cache-aside） public UserDto getByUuid(String userUuid) { return getShareCache(CACHE_USER_PROFILE, UserDto.class, userUuid, () -> loadFromDb(userUuid)); } // 更新后主动失效，避免脏读 public void updateProfile(String userUuid, UserDto dto) { saveToDb(userUuid, dto); removeShareCacheAll(CACHE_USER_PROFILE, UserDto.class, userUuid); } }
  • 命名建议：{domain}_{entity}_{scene}，如 usr_profile_detail。
  • Key 建议：单值 key 用业务唯一标识（如 userUuid）；多维条件使用复合 key（如 tenant:user:locale）。
  • 失效建议：写操作后优先 removeShareCacheAll，确保单值缓存和列表缓存一起失效。
  • 空值策略：热点空查询场景开启 null 缓存，其余场景默认关闭，避免占用过多缓存。

预发布检查清单（缓存专项）

  • 确认缓存名未与其他模块冲突，且命名可读。
  • 确认更新/删除流程都包含失效动作，不会只写库不清缓存。
  • 确认 Redis 开启/关闭两种模式下都能正常回退。
  • 确认 key 类型与 CacheConfig.key 一致（避免 String/Long 混用）。

性能调优建议（生产场景）

场景 风险 建议做法
缓存穿透（大量不存在 key） 频繁打到数据库 热点空数据开启 cacheNull/Null=true，并设置较短 TTL
缓存击穿（单热点 key 过期） 瞬时高并发回源 热点 key 提前续期，或在业务层增加单飞/互斥重建
缓存雪崩（同批 key 同时过期） 数据库突增 TTL 增加随机抖动，避免统一过期时间
跨实例短暂脏读 实例间本地缓存不一致 写后立即发布失效（当前已内置），并监控 Redis Pub/Sub 订阅状态
缓存 key 数量膨胀 内存上涨、淘汰变频繁 收敛 key 维度，按业务场景拆分缓存名并合理设置 max

TTL 分层建议（示例）

L1 热点且稳定数据：30~120 分钟（可叠加随机 0~15 分钟） L2 一般查询结果：10~30 分钟 L3 空值缓存：1~5 分钟（防穿透） L4 事务敏感数据：短 TTL + 写后强制失效

运维排障操作建议

  • 先看整体：GET /sys/cache/status、GET /sys/cache/list。
  • 再看目标缓存：GET /sys/cache/keys/{name} + GET /sys/cache/value/{name}?key=...。
  • 异常 key 精准清理：DELETE /sys/cache/key/{name}?key=...。
  • 批量异常时按缓存名清理：DELETE /sys/cache/clear/{name}，避免直接全量清空。
  • 仅在紧急场景执行：DELETE /sys/cache/clear/all。

监控指标与告警阈值建议

指标 建议阈值 触发后动作
缓存命中率（按缓存名） 连续 10 分钟低于 85% 排查 key 设计、TTL 过短、是否存在大量空值回源
数据库回源率 较日常基线提升 2 倍以上 检查是否出现击穿/雪崩，必要时临时延长热点缓存 TTL
Redis 缓存键增长率 30 分钟内增长超过 50% 检查 key 维度是否失控，必要时按缓存名清理并限流写入路径
缓存管理接口失败率（5xx） 5 分钟内超过 1% 优先检查 Redis 可用性、网络抖动、序列化异常
失效广播异常率（Pub/Sub） 出现持续错误日志或订阅中断 校验实例订阅状态，观察跨实例读取是否出现旧值

告警分级建议

  • P1：缓存雪崩导致核心接口大面积超时（需要立即降级/限流并批量修复 TTL）。
  • P2：单个核心缓存命中率骤降（需要 30 分钟内修复 key 或失效策略）。
  • P3：缓存键增长异常但业务未受损（需要当日完成容量与策略复盘）。

上线后一小时观察清单

  • 对比发布前后：命中率、回源率、接口 RT，确认无明显退化。
  • 抽查 1~2 个核心缓存名，确认 /sys/cache/keys/{name} 与实际业务量匹配。
  • 抽查更新链路：写操作后读取，确认没有跨实例陈旧数据。
  • 确认没有出现连续的 key 类型转换告警日志。

页面入口（templates）

  • 当前系统未提供独立缓存管理页面模板（templates/modules/sys/ 下无 cache.html）。
  • 缓存运维以接口方式提供，建议通过管理端二次封装页面或 API 调试工具调用。

权限码清单

无 @RequiresPermissions 权限码；接口采用控制器内权限判断： ShiroUtils.isLogin() && sysUserRoleService.isSystemAdministrator(userUuid)

最小可跑通用例

  1. 前置条件：使用系统管理员账号登录（非管理员会返回“无权限访问”）。
  2. 调用 GET /sys/cache/status，确认返回 redisEnabled、cacheCount。
  3. 调用 GET /sys/cache/list 选择一个缓存名（如 sysconfig）。
  4. 调用 GET /sys/cache/keys/sysconfig?page=1&size=20 获取键列表，再用 GET /sys/cache/value/sysconfig?key=... 查看值。
  5. 调用 DELETE /sys/cache/key/sysconfig?key=... 删除后再次查询，确认键已失效。

常见故障排查

现象 可能原因 处理建议
查询不到值但业务刚写入 写入实例与读取实例本地缓存尚未同步 确认 Redis Pub/Sub 可用，并检查 cache:invalidation 订阅日志
删除 key 无效 key 类型不匹配（如 Long 被当作 String） 核对缓存配置中的 keyType，必要时使用控制器转换规则
缓存穿透导致频繁回源 未开启 null 缓存 在配置中启用 Null=true 或在热点空数据场景开启 cacheNull
