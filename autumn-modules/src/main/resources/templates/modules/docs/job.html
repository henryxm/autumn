定时任务（job）

Autumn 提供两套调度能力：传统 schedulejob（cron 表达式）与接口式 LoopJob（固定周期接口）。在业务开发中，建议优先使用接口式任务，能显著降低表达式配置错误。

为什么推荐接口式定时任务（LoopJob）

方案 触发方式 常见风险 适用建议
cron 表达式任务 cronExpression 表达式写错、粒度难读、维护成本高 历史任务兼容、复杂时间规则
LoopJob 接口任务 实现 OneMinute/FiveMinute 等接口 基本无表达式错误风险 业务常规定时任务（推荐）
核心收益：用“接口语义”替代“字符串表达式”，把“写错 cron”问题前置消除。

选型决策图（先看这个）

是否是固定周期任务？ ├─ 是 -> 直接用 LoopJob 接口（OneMinute/FiveMinute/...） │ ├─ 需要防重入？-> @JobMeta(skipIfRunning = true) │ ├─ 需要超时观测？-> @JobMeta(timeout = xxx) │ └─ 多节点分配？-> assignTag + server.tag └─ 否 -> 用 schedulejob（cronExpression） ├─ 时间规则复杂（如每月/节假日）-> cron └─ 仍建议配置 run/pause/resume 与日志审计

接口周期速查表（LoopJob）

接口 典型用途 建议
OneSecond/ThreeSecond/FiveSecond 轻量状态轮询、快速缓冲刷新 必须幂等，关注 overrun 与并行开关
TenSecond/ThirtySecond 短周期同步、近实时汇总 优先开启防重入，避免任务堆积
OneMinute/FiveMinute/TenMinute 常规业务任务（推荐主力周期） 默认优先选择这一层级
ThirtyMinute/OneHour 报表聚合、缓存预热、对账 建议配置 timeout 和错误阈值
OneDay/OneWeek 归档、清理、周期报表 建议绑定服务器标签，避免多节点重复

功能入口

  • 任务管理：/job/schedulejob/*（save/update/delete/run/pause/resume）
  • 执行日志：/job/schedulejoblog/list、/job/schedulejoblog/toggle
  • 任务分配：/job/scheduleassign/list|save|update|delete
  • 接口式任务（统一管理）：/job/loop/*（stats/list/alerts/enable/disable/trigger/updateConfig 等）

接口式任务接入（推荐路径）

@Component @JobMeta( name = "用户缓存刷新任务", description = "每分钟刷新热点用户缓存", group = "user", skipIfRunning = true, timeout = 15000, maxConsecutiveErrors = 5, delay = 0, async = false ) public class UserCacheRefreshJob implements LoopJob.OneMinute { @Override public void onOneMinute() { // 业务逻辑 } }
  • 周期选择：直接实现 OneSecond/ThreeSecond/FiveSecond/TenSecond/ThirtySecond/OneMinute/.../OneWeek 对应接口。
  • 任务元信息：通过 @JobMeta 配置显示名、分组、超时、连续错误阈值、是否防重入、延迟与异步。
  • 调度注册：框架启动时自动扫描并注册，无需手写 cron 表达式。

统一管理能力（LoopJob）

  • 全局控制：/job/loop/pauseAll、/job/loop/resumeAll。
  • 任务控制：/job/loop/enable、/job/loop/disable、/job/loop/trigger。
  • 分类控制：/job/loop/enableCategory、/job/loop/disableCategory。
  • 配置管理：/job/loop/updateConfig（skipIfRunning/timeout/maxConsecutiveErrors/order 等）。
  • 服务器分配：/job/loop/updateAssign、/job/loop/assigns（配合 server.tag）。
  • 运行观测：/job/loop/stats、/job/loop/categories、/job/loop/list、/job/loop/alerts。
统一管理价值：同一套接口覆盖任务启停、告警、分配、统计，显著降低运维复杂度。

关键配置/参数

参数 示例 说明
cronExpression 0 */5 * * * ? 传统 cron 任务配置（接口式任务可不使用）
status 0/1 任务状态（暂停/运行）
skipIfRunning true 防重入，避免同一任务并发执行
timeout 15000 超时阈值（毫秒），用于慢任务告警
maxConsecutiveErrors 5 连续失败自动禁用阈值
assignTag / server.tag node-a / node-b 多节点任务分配，避免重复执行
parallelExecution true/false 同分类任务并行执行开关（秒级任务建议评估后启用）

上线前检查清单（任务专项）

  1. 确认任务是否必须用 cron；若固定周期优先改为 LoopJob 接口。
  2. 所有任务补齐幂等保障，避免重复执行副作用。
  3. 秒级/分钟级任务设置 skipIfRunning=true，并设置合理 timeout。
  4. 关键任务设置 maxConsecutiveErrors，并验证自动停用后的恢复流程。
  5. 多节点环境校验 assignTag 与 server.tag 匹配。
  6. 上线后观察 /job/loop/alerts 与分类 overrun 指标至少 1 个周期。

按业务类型选周期模板（可直接套用）

业务类型 推荐周期接口 建议参数
热点缓存刷新 OneMinute / FiveMinute skipIfRunning=true, timeout=5000~15000
支付/订单补偿轮询 ThirtySecond / OneMinute skipIfRunning=true, maxConsecutiveErrors=3~5
报表汇总/统计聚合 ThirtyMinute / OneHour timeout=30000+, 必须幂等
历史数据清理/归档 OneDay / OneWeek 绑定 assignTag，避免多节点重复执行
近实时轻量状态同步 ThreeSecond / FiveSecond 评估 parallelExecution，关注 overrun

回滚与降级策略（任务发布）

  • 发布后异常升高：先执行 /job/loop/disable 停掉单任务，避免扩大影响。
  • 系统级异常：执行 /job/loop/pauseAll，待恢复后 /job/loop/resumeAll。
  • 节点热点问题：临时调整 assignTag，把任务迁移到指定节点执行。
  • 性能瓶颈：先关闭并行执行（或反向启用）进行 A/B 观察，再决定长期配置。
建议为关键任务保留“手动触发补偿”能力，回滚后可通过 /job/loop/trigger 做定点修复。

最小可跑通用例

  1. 确认账号具备 job:schedulejob:list 和 job:schedulejob:run。
  2. 创建一个实现 LoopJob.OneMinute 的任务类并启动服务。
  3. 访问 GET /job/loop/list?category=OneMinute，确认任务被注册。
  4. 调用 POST /job/loop/trigger 手动触发，确认返回成功。
  5. 访问 GET /job/loop/stats 与 GET /job/loop/alerts，确认统计与告警可读。

故障排查模板

现象 可能原因 处理建议
任务未执行 任务被禁用、分类被禁用、全局暂停或分配标签不匹配 依次检查 globalPaused/categoryEnabled/enabled/assignTag
任务重复执行 多节点都在执行或未开启防重入 设置 skipIfRunning=true 并校验 assignTag/server.tag
任务自动被停用 连续错误超过阈值 检查 maxConsecutiveErrors 和错误日志，修复后执行 /job/loop/enable
秒级任务频繁超限 批量耗时大于分类间隔 查看 /job/loop/categories 的 overrun 指标，必要时启用并行执行
