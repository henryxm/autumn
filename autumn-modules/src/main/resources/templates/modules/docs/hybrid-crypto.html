混合加解密（RSA + AES）

该能力由 RsaController、RsaService、AesService 共同实现。核心目标是：用 RSA 做密钥交换，用 AES 做业务数据加密，兼顾安全与性能。

设计思路

  • RSA 负责：服务端密钥对生成、客户端公钥管理、AES 密钥安全下发。
  • AES 负责：业务报文高频加解密（性能优于纯 RSA）。
  • 通过 session 关联一套密钥上下文，支持密钥过期与缓冲期平滑切换。
推荐优先使用 /rsa/api/v1/init 一次性握手接口，可减少客户端交互次数。

核心接口

  • POST /rsa/api/v1/public-key：获取服务端 RSA 公钥。
  • POST /rsa/api/v1/client/public-key：上传客户端 RSA 公钥。
  • POST /rsa/api/v1/aes-key：获取用客户端公钥加密后的 AES key/vector。
  • POST /rsa/api/v1/init：合并握手（服务端公钥 + 客户端公钥登记 + AES 密钥下发）。
  • GET/POST /rsa/api/v1/endpoints：获取加密端点描述（由服务端扫描）。

推荐接入流程（客户端）

  1. 客户端生成本地 RSA 密钥对，并生成唯一 session。
  2. 调用 /rsa/api/v1/init 提交 session + clientPublicKey。
  3. 从返回中拿到：
    • 服务端公钥（后续客户端可用来加密上行敏感内容）
    • 加密后的 AES key/vector（用客户端私钥解密）
    • 加密端点描述（用于判定哪些接口要走加密）
  4. 业务请求改为 AES 加密报文，服务端使用同一 session 解密。
  5. 若返回密钥相关错误码，重新执行握手流程。

触发条件（请求解密 / 响应加密）

  • 请求解密（EncryptArgumentResolver.resolveArgument）：仅当请求体可解析为 Encrypt，且 ciphertext 与 session 都非空时才执行解密。
  • 请求不解密的情况：缺少 ciphertext 或缺少 session 时，不会解密，保留原始对象继续流程。
  • 响应加密（EncryptInterceptor.beforeBodyWrite）：当请求头包含 X-Encrypt-Session 时进入加密流程。
  • 返回值兼容增强：若返回体不是 Encrypt，且是 JSON 响应，会先自动包装为 Response.ok(body) 再加密。
  • 响应不加密的情况：缺少 X-Encrypt-Session 时直接返回原始 body；/rsa/* 接口显式排除，不做响应加密；非 JSON 响应（文件/文本）不做自动包装。
  • 算法选择：响应侧读取 X-Encrypt-Algorithm，默认 AES；若为 RSA 则走 RSA 加密。
关键判断不是“所有请求都解密/所有响应都加密”，而是“是否携带加密上下文（session/ciphertext/header）”。

force 约束说明

  • 请求侧：解析器中存在 force 校验逻辑，开启后会强制要求 ciphertext 与 session 不可为空。
  • 响应侧：拦截器检查方法上的 @Endpoint(force=true)，若未携带 X-Encrypt-Session，返回 FORCE_ENCRYPT_SESSION_REQUIRED。

老接口兼容方案（同接口支持明文 + 加密）

当存量客户端仍走旧明文协议，而新客户端要走加密协议时，建议在请求参数层做兼容，而不是拆两套接口。

  • 新客户端：使用 Request<T> 包装业务数据，按框架规则携带 ciphertext/session。
  • 旧客户端：仍提交旧的明文字段。
  • 服务端：统一接收 cn.org.autumn.model.CompatibleRequest<T>，优先取 data（新协议），否则回退到旧字段映射。
  • 返回值：统一使用 Response<T>，由拦截器依据 X-Encrypt-Session 决定是否加密返回体。
// 示例（xsign）：同一个 hyper 接口同时兼容新旧协议 @RequestMapping(value = "hyper", method = RequestMethod.POST) public Response<Hyper> hyper(@RequestBody(required = false) CompatibleRequest<Hyper> request, HttpServletRequest servlet) { Hyper hyper = request != null ? request.resolveBody(Hyper.class, gson) : null; if (hyper == null) return Response.error("非法请求"); Hyper ex = xSignChaoCodeService.exchange(hyper); return ex == null ? Response.error("无效兑换码") : Response.ok(ex); }

混合模式组合分析（请求/返回）

请求参数形态 返回值形态 行为结果
Request<T>（含密文） Response<X> 请求解密 + 响应加密（标准推荐路径）
CompatibleRequest<T>（密文或明文） Response<X> 请求兼容解析 + 响应按 header 加密
老 DTO（非 Request） Response<X> 请求侧无法自动解密；响应可按 header 加密
CompatibleRequest<T> 老返回对象（非 Encrypt） 若带 X-Encrypt-Session 且为 JSON，自动包装后加密
任意请求 文件/非 JSON 响应 不做自动包装，保持原返回
你提到的“请求不用 Request 但返回用 Response”的组合是支持的：请求不自动解密，响应仍可依据 X-Encrypt-Session 加密。

控制器改造最小模板（3 类接口）

下面给出可直接落地的最小改造方式，优先保证“兼容老客户端 + 支持新加密客户端”，并减少一次性大改风险。

// 模板A：推荐新接口（请求兼容 + 返回标准） @PostMapping("/biz/create") public Response<BizVO> create(@RequestBody(required = false) CompatibleRequest<BizDTO> request) { BizDTO dto = request != null ? request.resolveBody(BizDTO.class, gson) : null; if (dto == null) return Response.error("非法请求"); return Response.ok(service.create(dto)); }
// 模板B：老请求对象保留（不走自动解密）+ 返回标准Response @PostMapping("/biz/update") public Response<Boolean> update(@RequestBody BizDTO dto) { if (dto == null) return Response.error("参数为空"); return Response.ok(service.update(dto)); }
// 模板C：老返回对象保留（拦截器会在有X-Encrypt-Session且JSON时自动包装再加密） @PostMapping("/biz/get") public BizVO get(@RequestBody(required = false) CompatibleRequest<BizQuery> request) { BizQuery query = request != null ? request.resolveBody(BizQuery.class, gson) : null; return service.get(query); }
  • 优先级建议：优先改“高频写接口”和“敏感数据接口”，先用模板A；低风险读接口可先用模板C过渡。
  • 一致性建议：最终目标是全部收敛到模板A（CompatibleRequest<T> + Response<T>）。

批量迁移清单（按控制器执行）

  1. 扫描控制器，分组标记：Request/CompatibleRequest 参数、老 DTO 参数、Response 返回、老对象返回。
  2. 先改“入参复杂 + 返回敏感”的接口：参数改 CompatibleRequest<T>，返回改 Response<T>。
  3. 保留老返回对象的接口，验证带 X-Encrypt-Session 时是否已自动包装加密。
  4. 对下载/导出接口单独核对，确认其 Content-Type 为非 JSON，避免错误包装。
  5. 逐批灰度，每批完成后回归“明文请求 + 加密请求”两套路径。

验收用例（每个接口至少覆盖）

  • 明文请求 + 无 X-Encrypt-Session：结果应保持历史行为。
  • 加密请求（含 ciphertext/session）+ 有 X-Encrypt-Session：应正确解密并加密返回。
  • 明文请求 + 有 X-Encrypt-Session：请求不解密，返回按规则加密（或自动包装后加密）。
  • 文件下载接口 + 有 X-Encrypt-Session：不应被包装为 Response。
  • @Endpoint(force=true) 接口缺少 session：应返回 FORCE_ENCRYPT_SESSION_REQUIRED。

请求示例

POST /rsa/api/v1/init { "session": "device-001", "data": { "publicKey": "-----BEGIN PUBLIC KEY-----...-----END PUBLIC KEY-----", "expireTime": 1760000000000 } }
响应（结构示意） { "code": 0, "data": { "ras": {"publicKey":"...", "session":"device-001", "expireTime": ...}, "aes": {"key":"", "vector":"", "session":"device-001", "expireTime": ...}, "endpoints": "" } }

关键错误码（常见）

  • RSA_CLIENT_PUBLIC_KEY_NOT_FOUND (1005)：客户端公钥不存在或过期。
  • RSA_KEY_FORMAT_ERROR (1007)：RSA 密钥格式错误。
  • RSA_SESSION_REQUIRED (1009)：session 为空。
  • AES_KEY_NOT_FOUND (1100)：AES 密钥不存在或过期。
  • AES_KEY_FORMAT_ERROR (1105)：AES key/vector 格式异常。

实现细节说明

  • RsaService.getKey() 与 AesService.getKey() 都带“临期重建”逻辑（客户端缓冲时间）。
  • encrypt/decrypt 过程优先使用现有缓存，不在高频路径主动刷新密钥，避免客户端与服务端密钥不同步。
  • 服务端配置了“有效期 + 服务端冗余保留时间”，用于密钥轮换期间平滑解密。

安全使用建议

  • 每个终端/会话使用独立 session，不要多个客户端复用。
  • 客户端私钥只保存在本地安全区，不出端。
  • 检测到 1005/1100/1003/1103 等错误时，立即重新握手。
  • 生产环境建议启用 HTTPS，与报文加密形成双层保护。
