授权与扫码登录流程指南

本页从业务视角说明 Autumn 中「授权登录 + 网站接入 + 扫码授权」的整体架构、数据关系与四种典型使用场景。适合运维、后端与对接方快速理解「为什么要配应用」「扫码如何变成登录/授权码」。

相关模块：cn.org.autumn.modules.oauth（OAuth2 授权服务器）、 cn.org.autumn.modules.client（网站 OAuth 客户端与合并绑定）、 cn.org.autumn.modules.qrc（扫码编排层）。 仓库详细 API 见 docs/AI_QRC.md、docs/AI_QRC_API.md。

1. 能力总览

Autumn 同时扮演两类角色，并可被第三方调用：

角色含义典型入口
授权服务器（AS）本系统签发授权码 / 访问令牌，管理「谁可以来拿 token」/oauth2/authorize、/oauth2/token
网站客户端本网站作为 OAuth Client，向 AS 发起登录并换 token/client/oauth2/callback
扫码编排层（QRC）用「手机 App 扫码 + 确认」代替 PC 输入密码，或代替 OAuth 授权页点同意/qrc/web/v1/*、/qrc/api/v1/*

扫码不是独立的认证协议，而是编排层：最终仍落到 Shiro Session（同源网站登录）或 OAuth 授权码（第三方授权）。

2. 为什么要创建应用

运维口中的「应用」或「客户端」，在库里对应四块配置 + 一张审计表。它们通过 clientId（应用标识，如域名 app.example.com）逻辑关联，没有数据库外键，由「授权客户端中心」或域名初始化时一并创建。

授权登记

oauth_client_details

登记谁可以向本系统申请授权：应用标识、密钥、回调地址、授权范围、是否可信。第三方 App / 本站域名作为 OAuth Client 时都需要这条记录。

网站接入

client_web_authentication

登记本网站 outbound 登录要连到哪里：授权页地址、令牌地址、用户资料地址。用于「本站用 OAuth 登录本站」这类场景。

绑定关系

client_web_oauth_combine

把「授权登记」与「网站接入」绑在同一 clientId 下。常见做法：一个域名 = 一套 AS 侧 client + 一套 Web 侧 client，合并后运维只需记一个标识。

扫码策略

qrc_client_grant

控制该应用是否允许扫码、扫码完成后如何交付结果（轮询授权码 / 轮询令牌 / 应用跳转 / 回调通知）、是否强制手机二次确认等。

创建方式

方式触发产生数据
域名自动创建 插入 client_web_oauth_combine（如站点初始化） 同时生成授权登记 + 网站接入 + 合并行；回调默认为 {站点}/client/oauth2/callback
授权客户端中心 运维页「新建」→ POST /client/authhub/create 同上 + 默认扫码策略行
分散维护 各模块 CRUD 页单独维护 需自行保证 clientId 一致；Hub 保存时会尝试补全合并关系
可信应用：OAuth 类扫码（授权 / 确认 / 设备码）要求授权登记中 trusted=1，且回调地址与登记完全一致，否则拒绝发码。

6. 场景 A：网站扫码登录（SELF_WEB_LOGIN）

目的：用户已在手机 App 登录，PC 网站不想输密码，扫二维码后在 PC 建立与 App 同一用户的 Session。

Intent：SELF_WEB_LOGIN · Handler：SelfWebLoginHandler

  1. PC 调用 POST /qrc/web/v1/ticket/create，body 含 intent: SELF_WEB_LOGIN，获得票据 uuid 与 qrUrl。
  2. PC 展示二维码（内容为 {host}/qrc/v1/t/{uuid} 或 App 协议链接）。
  3. App 解析 uuid，带用户 Token 调用 POST /qrc/api/v1/ticket/scan，再 POST .../confirm。
  4. 服务端 生成一次性 exchange 令牌（默认 TTL 约 60s，见 QRC_CONFIG）。
  5. PC 轮询 GET /qrc/web/v1/ticket/status?uuid=...，状态为 CONFIRMED 时拿到 exchange。
  6. PC 调用 POST /qrc/web/v1/session/exchange，Shiro 建立 Session，跳转首页。
扫码 ≠ 登录：必须 App 端 confirm；exchange 仅用一次。登录审计 way=qrc_self。

7. 场景 B：第三方授权扫码（OAUTH_AUTHORIZE）

目的：第三方浏览器打开 /oauth2/authorize?client_id&redirect_uri&scope&state，用户未在 AS 登录；优先展示扫码页，App 确认后第三方获得 code。

Intent：OAUTH_AUTHORIZE · 入口：AuthorizationController + login.html（mode=authorize）

  1. 第三方将用户重定向到 AS 授权 URL（须 trusted client + redirect_uri 完全匹配）。
  2. AS 检测未登录且全局 oauthQrFirst=true（QRC_CONFIG）→ 创建 OAUTH_AUTHORIZE 票据 → 渲染 login.html 扫码 Tab。
  3. PC 轮询票据；App scan + confirm。
  4. ClientGrantService.deliverOAuth 签发 authorization code，拼装 redirect 或返回 result.code。
  5. PC 跳转 redirect_uri?code=...&state=...；第三方后端 POST /oauth2/token 换 access_token。
建票失败时降级为账号密码 Tab，不会静默发 code。与 consent 场景（失败直接报错）策略不同。

8. 场景 C：授权二次确认（OAUTH_CONSENT）

目的：浏览器已在 AS 有 Session，但策略要求「手机再确认一次 scope」（防会话劫持冒点同意）。

触发：qrc_client_grant.consent = true（默认 false）

  1. 用户访问 /oauth2/authorize，AS 已登录 → requireAppConsent(clientId) 为 true。
  2. 创建 OAUTH_CONSENT 票据，展示 consent 模式登录页。
  3. App 扫码 confirm 后同样 deliverOAuth 发 code 并 redirect。
  4. 若建票失败 → 返回 OAuth 错误，不发 code。

9. 场景 D：设备码 / 自建二维码（OAUTH_DEVICE）

目的：第三方自有 UI（电视、CLI、Partner 后台），不经过 AS 的 login.html，自行展示 QR 并轮询或收 Webhook。

Intent：OAUTH_DEVICE · 入口：POST /qrc/open/v1/ticket/create（client_id + client_secret）

  1. 第三方服务端带密钥创建票据，获得 uuid / qrUrl，在自有界面展示 QR。
  2. App scan + confirm（与 B/C 相同）。
  3. 按扫码策略 delivery 交付结果：轮询拿 code/token、Webhook 推送、或 Deep Link 回 App。
  4. 第三方用 code 调 /oauth2/token 或直接使用返回的 token（视 delivery 而定）。

10. 扫码票据状态机

活跃票据存 Redis/内存，键 qrc:ticket:{uuid}，迁移带分布式锁。

PENDING 待扫码 → SCANNED 已扫码 → CONFIRMED 已确认 → COMPLETED 已完成
DENIED 已拒绝 CANCELLED 已取消 EXPIRED 已过期
Intent用途说明
SELF_WEB_LOGIN同源 PC 网站登录
OAUTH_AUTHORIZE未登录用户的 OAuth 授权
OAUTH_CONSENT已登录 + 手机二次确认
OAUTH_DEVICE第三方 Open API 自建 QR

11. 结果交付方式（扫码策略 · 结果方式）

模式适用场景PC/第三方如何拿到结果
POLL_CODE默认；OAuth 授权码模式轮询 status，result.code + redirectUri
POLL_TOKEN希望轮询直接拿 token轮询 result.accessToken 等
DEEP_LINK结果回跳到原生 AppApp 获得 deepLink；需配置 schemes 白名单
WEBHOOK服务端异步通知POST 到策略中的通知地址，带签名密钥

13. API 速查

PC Web（匿名）

POST /qrc/web/v1/ticket/create
GET /qrc/web/v1/ticket/status?uuid=...
POST /qrc/web/v1/session/exchange

手机 App（须用户 Token）

GET /qrc/api/v1/ticket/{uuid}
POST /qrc/api/v1/ticket/scan
POST /qrc/api/v1/ticket/confirm
POST /qrc/api/v1/ticket/deny

第三方 Open API（client_id + secret）

POST /qrc/open/v1/ticket/create
GET /qrc/open/v1/ticket/status?uuid=...&clientId=...&clientSecret=...

OAuth2 AS

GET /oauth2/authorize?response_type=code&client_id=...&redirect_uri=...&scope=...&state=...
POST /oauth2/token
GET /oauth2/userInfo

Hub 管理 API

GET /client/authhub/list
GET /client/authhub/info/{clientId}
POST /client/authhub/create | save | delete
GET /client/authhub/tickets/{clientId}

QR 链接（公开字段，无密钥）

GET /qrc/v1/t/{uuid} → { uuid, intent, status }
